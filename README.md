# Partsone バックエンド研修課題リポジトリ

本リポジトリは、バックエンド育成プログラム「デモアプリの開発」における基本実装（コメントのCRUD機能およびAPI設計）の成果物です。

## 📌 実行・環境構築手順
APIの動作確認およびテストデータを用いた検証を行うための手順です。

### 1. データベースのセットアップ（テストデータの作成）
APIのテストを行うために、ダミーのコメントデータを作成するシーダーを用意しています。環境構築後、以下のコマンドを実行してデータベースをリセットし、テストデータを挿入してください。

```bash
# コンテナの立ち上げ（まだ起動していない場合）
./sail up -d

# データベースの初期化とシーダー（テストデータ）の実行
./sail artisan migrate:fresh --seed
```

### 2. APIドキュメント (Swagger) の確認方法
本課題のAPI設計書はSwaggerを使用して作成しています。以下の手順でAPI仕様を確認・テストすることができます。

1. プロジェクトのルートディレクトリで以下を実行してキャッシュをクリアしてください:
   `./sail artisan optimize:clear`
2. ブラウザで以下のURLにアクセスしてください:
   `http://localhost:8002/`

*※注: 環境構築時、`.env` ファイルの `OPENAPI_FILE_NAME` が実際の設定ファイル（`api-docs.yml`）と一致しているかご確認ください。*

## 2. 取り組んだ課題ごとの工夫点・アピールポイント
### 評価（Rating）システムの独自導入
単なるテキストのコメント機能に加え、1〜5段階の `rating`（評価）カラムを独自に追加実装しました。これにより、将来的に「評価の高い記事をランキング表示する」といった拡張が可能になります。また、システムに評価機能が存在することで、記事作成者が読者の反応を意識し、より質の高いコンテンツを生み出すというポジティブな競争が生まれるようなビジネスロジックを想定して設計しました。

### 調査・解決したこと（API設計書の作成について）
開発を進める中で、API設計書の作成において提供されたテンプレートの項目（リソース名や多階層のレスポンス定義など）をどのように今回のMVPに落とし込むか悩みました。
調査や質問を通して、今回のようなシンプルなCRUD APIにおいては、複雑なExcelシートよりもMarkdownの表やSwagger UIを中心としたドキュメント管理が、バージョン管理（Git）や保守性の観点から現場のベストプラクティスに近いことを学びました。そのため、指定の仕様を満たしつつ、よりシンプルで「人に理解してもらいやすい」形式に調整してドキュメントを作成しました。

### 可読性と開発体験への配慮
「人に理解してもらう」コードを目指し、以下の点を意識しました。
*   **命名規則と規約：** 変数名は誰が見ても用途がわかる明確な英単語を使用し、PHPの標準的なコーディング規約（PSR）を意識してフォーマットを整えました。
*   **コメントの活用：** コマンドラインからファイルを作成した際に自動生成されるPHPDocなどのコメントブロックを適切に残し、コントローラー内の処理の意図が伝わりやすいようにしています。

## 3. 注意して見てほしい点
コメント機能のメインとなる**CRUD処理（GET, POST, PUT, DELETE）のルーティングとコントローラーのロジック**です。
Swagger UIを用いて、全てのAPIが仕様通り（文字数制限のバリデーションや、最新順のソートなど）に動作することを念入りにテスト・確認いたしました。

## 4. 得た学び・感想
今回が初めてのDockerの利用でしたが、環境構築におけるコンテナ技術の圧倒的な有用性を学ぶことができました。普段とは別のWSL環境を用意して取り組んだため、初期段階で権限（パーミッション）周りのエラーに直面しましたが、ネットの文献調査に加え、生成AIをデバッグの壁打ち相手として積極的に活用することで、自力で原因を特定しエラーを解決する力を養うことができました。

また、これまでアプリケーション開発の経験はありましたが、Webシステムの構造やAPIをゼロから構築するのは今回が初めてでした。最初はPHPやLaravelの概念に圧倒され、提供いただいた教材（ドットインストール等）に加え、Webの仕組みをより深く理解するためにMVCやBlade（テンプレートエンジン）を一から解説する外部チュートリアルも併用して学習を進めました。

Webの全体像が掴めてきた段階で、「今回の課題はBladeで画面を作るのではなく、APIとしてデータを提供する要件である」ということに気づき、画面の代わりにSwaggerを使ってAPIの動作を確認する手法へと実装の舵を切りました。Laravelへの理解が深まるにつれて、最初はよく分からなかったAPI設計書の各項目の意味（リソース名や構造など）がはっきりと理解できるようになり、点と点が繋がる非常に楽しい開発体験となりました。
